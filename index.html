Logic Explanation

The Loot Array Is the Source of Truth

The lootItems array is declared at the very top of script.js, outside of every function, so it persists across every click instead of resetting. The screen never decides what the totals are. It only displays whatever is currently sitting in that array. If I want to change what's on screen, I change the array first and let updateUI() catch up. Buttons trigger state changes. They don't calculate anything themselves.

Why Quantity Was Added

Phase 1 treated each loot entry as a single object with a flat value. Phase 2 requires total value to be calculated as value times quantity, so each loot object now stores name, value, and quantity as plain properties. This lets a single stack of arrows or potions be represented as one entry instead of forcing duplicate rows for identical items.

Where updateUI() Does Its Work

updateUI() is the only place where totals are calculated and the only place where the DOM gets touched for rendering. It walks the array with a traditional for loop to sum value * quantity for every item, rebuilds the loot table rows, recalculates the per-member split using current party size, and then decides what should be visible or disabled. Nothing outside updateUI() performs a calculation. addLoot(), removeLoot(), and the party size listener only mutate state and then call updateUI().

State Controls Visibility and Enabling

updateUI() checks whether the array is empty and whether party size is a valid number 1 or greater. Based on that, it toggles a .hidden class on the empty message and the results area, and sets the disabled attribute on the Split Loot button. No inline styles are touched directly in JavaScript. Visibility is always a reflection of state, never something set independently of it.

Where Validation Protects State

addLoot() checks the name, value, and quantity before anything is pushed into the array, and returns early with an error message the moment one check fails. The party size input is validated inside updateUI() itself before it's used in the split calculation, so an invalid party size simply causes the results to hide rather than producing a broken number on screen. Invalid data never reaches the array, and invalid party size never reaches the math.
